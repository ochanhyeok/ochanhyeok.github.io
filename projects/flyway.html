Resilience

SafeCount 장애 격리

9개 메트릭 각각 독립 조회. 1개 실패해도 나머지 8개 정상 표시.

Key Decisions

01

Atomic SQL로 로그인 실패 처리 (Race Condition 방지)

Problem

로그인 실패 시 카운트 증가 + 5회 시 잠금이 필요. Java에서 SELECT → UPDATE 순서로 처리하면 동시 시도 시 race condition으로 실패 횟수가 누락될 수 있다.

Decision

단일 UPDATE문에 CASE 표현식으로 카운트 증가 + 조건부 잠금을 원자적 처리. SELECT 없이 UPDATE만으로 완결.

Result

네트워크 왕복 1회, 락 경합 없음, 5회 실패 시 정확히 30초 잠금 보장.

UPDATE admin
SET failed_login_count = COALESCE(failed_login_count, 0) + 1,
    locked_until = CASE
        WHEN COALESCE(failed_login_count, 0) + 1 >= 5
        THEN #{lockedUntil}
        ELSE locked_until
    END,
    updated_at = NOW()
WHERE admin_id = #{adminId}

02

SafeCount 패턴으로 대시보드 장애 격리

Problem

대시보드가 9개 메트릭을 조회하는데, 하나의 집계 쿼리가 실패하면 전체 대시보드가 렌더링 안 됨.

Decision

CountSupplier 함수형 인터페이스 + safeCount() 래퍼로 각 메트릭을 독립 감싸. 코드 중복 없이 부분 실패 허용.

Result

9개 중 1개 DB 에러여도 나머지 8개 정상 표시. 실패한 메트릭만 0 반환.

private long safeCount(CountSupplier supplier) {
    try {
        return supplier.get();
    } catch (Exception e) {
        log.warn("Count query failed, returning 0", e);
        return 0;
    }
}

// 9개 메트릭 각각 독립 실패 허용
.dailyVisitors(safeCount(dashboardRepository::countDailyVisitors))
.dailyPayments(safeCount(dashboardRepository::countDailyPayments))

03

항공편 조회 쿼리 최적화 (19초 → 0.019초, 1000배)

Problem

관리자 특가관리 페이지 진입 시 항공편 목록 조회에 19초 소요. EXPLAIN 분석 결과 flight_number_aircraft_map(622행)과 flight(183,769행) JOIN이 병목.

Decision

프론트엔드 코드를 확인하여 airline 정보가 미사용임을 확인한 후, 2개의 LEFT JOIN을 제거. 추가로 커버링 인덱스 생성.

Result

핵심 교훈: 인덱스 추가보다 불필요한 JOIN 제거가 더 효과적일 수 있다.

조회 시간19초→0.019초
스캔 행183,769→~100
JOIN 수2개→0개

04

JWT type claim으로 Cross-Token 위조 방지

Problem

User JWT와 Admin JWT가 공존. 서명 키 하나가 유출되거나 실수로 같은 키를 사용하면 User 토큰으로 Admin API에 접근 가능.

Decision

별도 서명 키 + "type": "ADMIN" claim 이중 방어. 검증 시 type 불일치면 무조건 거부.

Result

서명 키 분리 실패 시에도 cross-token 공격 차단. 검증 코드 2줄 추가로 보안 계층 추가.

// 생성
.claim("type", "ADMIN")

// 검증
String type = claims.get("type", String.class);
if (!"ADMIN".equals(type)) {
    return false;
}

05

INSERT IGNORE + Event-Driven 방문자 추적

Problem

방문 추적에서 (1) 동일 세션 중복 방지, (2) 추적 실패가 사용자 요청 블로킹하면 안 됨, (3) 동시 요청 race condition 방지 필요.

Decision

3중 방어: 세션 캐시 → DB existsToday → INSERT IGNORE. Spring Event로 디커플링 + REQUIRES_NEW 독립 트랜잭션.

Result

중복 기록 0건, 추적 실패 시 사용자 요청 정상 진행 보장, 서버 확장 시에도 DB 레벨 중복 방지 유지.

Troubleshooting

ConcurrentHashMap NPE — WebSocket 핸드셰이크 실패

증상: 대시보드 진입 시 WebSocket 연결 간헐적 실패, NullPointerException at ConcurrentHashMap.putVal.

원인: HTTP 세션의 adminName, adminRole 속성이 없는 경우 null 반환. ConcurrentHashMap은 null 값을 허용하지 않아 NPE 발생.

해결: null 체크 추가. 값이 있을 때만 attributes에 put.

교훈: HashMap에서 ConcurrentHashMap으로 변경 시 null 처리가 달라진다. ConcurrentHashMap은 key/value 모두 null 불허.

방문자 상세 조회 SQL 컬럼 불일치

증상: "오늘 방문자" 목록 조회 시 Unknown column 'up.name' SQL 에러.

원인: user_profile 테이블에 name 컬럼이 없고 first_name + last_name으로 분리. 다른 팀원이 설계한 스키마를 잘못 참조.

해결: CONCAT + COALESCE로 null-safe 이름 조합, NULLIF(TRIM(...))로 빈 문자열 처리.

교훈: 다른 팀원 테이블을 JOIN할 때 반드시 실제 DDL을 확인. name vs first_name + last_name 같은 차이는 사전 합의 필요.

세션 고정 공격(Session Fixation) 방지 — CodeRabbit 리뷰

증상: CodeRabbit AI 리뷰에서 로그인 시 기존 세션 재사용으로 세션 고정 공격 취약 지적.

원인: 로그인 전 JSESSIONID가 로그인 후에도 유지되면, 공격자가 미리 심어둔 세션 ID로 관리자 권한 탈취 가능.

해결: 로그인 성공 시 oldSession.invalidate() → getSession(true)로 JSESSIONID 갱신.

교훈: 인증 성공 시 세션 ID를 반드시 갱신. AI 코드 리뷰(CodeRabbit) 도입으로 보안 취약점을 사전 발견 가능.

Security FilterChain 충돌 — Admin 로그인이 안 되는 문제

증상: Admin 로그인 시 인증이 계속 풀리거나, 아예 /admin 경로 접근이 차단됨.

원인: 팀원이 구현한 SecurityConfigWeb(@Order(3))의 anyRequest().authenticated()가 /admin/** 경로까지 가로채서 JWT 인증을 강제. Admin은 세션 기반인데 JWT 필터가 먼저 동작하여 인증 실패.

해결: Spring Security FilterChain을 3개로 분리. SecurityConfigApi(@Order(1), /api/**), AdminSecurityConfig(@Order(2), /admin/**), SecurityConfigWeb(@Order(3), 나머지). 각각 antMatcher로 경로를 명시하여 범위를 격리.

교훈: Spring Security에서 다중 인증 방식(세션 vs JWT)이 공존할 때, FilterChain의 @Order와 antMatcher 범위를 명확히 분리해야 한다. 특히 anyRequest()는 의도치 않은 경로까지 잡을 수 있다.

로컬 서버 재시작 시 방문자 중복 카운트

증상: 개발 중 방문자 수가 비정상적으로 증가. 팀원들이 로컬 서버를 껐다 킬 때마다 새로운 세션이 생성되면서 같은 사람이 여러 번 카운트됨.

원인: 세션 기반 1차 캐시(visitorTracked)는 서버 재시작 시 초기화. DB 중복 체크는 session_id 기반인데, 서버 재시작 시 새 JSESSIONID가 발급되어 동일 사용자가 새 방문자로 기록.

해결: INSERT IGNORE로 DB 레벨 중복 방지를 최종 방어선으로 설정. 세션 캐시는 성능 최적화(DB 조회 감소)용이고, 진짜 중복 방지는 DB의 unique 제약이 담당하는 구조로 정리.

교훈: 세션 기반 캐시는 서버 라이프사이클에 종속된다. 중복 방지의 최종 보장은 항상 DB 레벨에서 해야 한다.

Infrastructure — AWS 배포

Build

Maven WAR 패키징

Spring Legacy + Java 17 → flyway.war 생성. 프로파일 분리(dev/prod)로 환경별 설정.

Transfer

S3 경유 또는 SCP 직접 전송

aws s3 cp로 S3 버킷 업로드 → EC2에서 다운로드. 또는 scp로 직접 전송.

Deploy

EC2 + 외부 Tomcat + Nginx

Spring Boot와 달리 Legacy는 내장 Tomcat이 없어 EC2에 Tomcat을 직접 설치·관리해야 했다. JDK 설치 → Tomcat 설치·설정(setenv.sh 힙 메모리) → WAR 배포 → Nginx 리버스 프록시 설정까지 인프라를 직접 구성. 배포마다 Tomcat stop → WAR 교체 → start → 로그 확인 과정을 수동으로 진행하며 배포 파이프라인의 필요성을 체감.

Database

AWS RDS (MariaDB)

RDS 엔드포인트로 연결. EC2 보안 그룹에서 RDS 접근 허용.

Troubleshooting

EC2 서버가 갑자기 꺼지는 문제 (OOM Kill)

Problem

t2.micro(1GB RAM)에서 Tomcat이 메모리를 혼자 1GB 점유하여 OS가 숨 쉴 공간이 없었다. 메모리 부족 시 Linux의 OOM Killer가 Tomcat 프로세스를 강제 종료 — 서버가 갑자기 꺼지는 현상 반복.

Solution

1. 시간 동기화 (UTC → KST) JVM 시간을 한국 시간으로 변경하여 디버깅 시 시간 계산 불필요
2. 메모리 다이어트 (1GB → 512MB) Tomcat 힙을 -Xmx512m으로 절반으로 줄여 OS에 400MB+ 여유 확보
3. Swap 2GB 설정 디스크를 가상 메모리로 활용하여 메모리 부족 시에도 OOM Kill 방지

Lesson

Free Tier(t2.micro)처럼 리소스가 제한된 환경에서는 애플리케이션이 OS와 메모리를 공유한다는 점을 인식해야 한다. Spring Boot는 내장 Tomcat이라 기본 힙이 적당하지만, 외부 Tomcat은 기본값이 크게 잡히므로 명시적으로 조절해야 한다.

Team Impact

  • 프로젝트 초기 세팅 전체 (하루 소요) — Spring Boot와 달리 Legacy는 pom.xml 의존성 버전 호환성을 하나하나 확인하고, web.xml/root-context.xml/servlet-context.xml을 직접 구성해야 했다. Maven reload → TestController 실행 → 검증을 반복하며, 팀원들이 환경설정 없이 바로 개발할 수 있도록 세팅. 이 과정에서 Spring Boot가 자동으로 해주는 것들(설정 통합, 내장 Tomcat, 의존성 관리)의 가치를 체감.
  • ApiResponse<T> 표준화 — success(), error(), success(data, page) 정적 팩토리 메서드로 팀 전체 REST API 응답 통일.
  • ErrorCode 체계 설계 — 모듈별 접두사(C=공통, U=회원, A=관리자, P=결제, R=예약) 기반. 에러 코드 충돌 없이 각 팀원이 독립적으로 추가 가능.
  • Legacy 환경설정 가이드 작성 — IntelliJ에서 Spring Legacy + Tomcat 환경 구성법을 스크린샷 포함 노션 문서로 작성. 팀원들이 각자 CRUD 연습할 때 이 문서 보고 환경 구성. 노션 문서
