Overview

동아리 운영을 위한 출석 관리·게시판·모의투자·AI 퀀트봇을 통합한 웹플랫폼. 나는 출석(Attendance) 모듈 전체를 담당했다 — Session/Round/Attendance 3계층 엔티티 설계, GPS 기반 위치 검증, 체크인 플로우, 테스트 코드까지 약 4,000줄, 34개 파일.

이 프로젝트는 첫 팀 프로젝트였다. 기술적으로 대단한 걸 했다기보다, 여기서 경험하고 배운 것들이 이후 WeFin과 Flyway에서 내가 주도적으로 도입한 것들의 기반이 되었다.

Key Decision — Haversine 공식 기반 GPS 출석 검증

Problem

출석 시 위치 검증이 필요하지만, 동아리 프로젝트라 외부 API 비용을 쓸 수 없었다.

Decision

출석 검증에 필요한 건 "두 좌표 간 거리가 반경 내인가" 하나뿐이므로, Haversine 공식을 @Embeddable Location에 직접 구현. Google Maps API나 PostGIS 없이 외부 의존 0.

Result

LocationTest에서 서울-부산 거리 검증 통과. isWithinRange()에서 radiusMeters가 null이면 검증 비활성화 (graceful degradation).

// Location.java — Haversine 거리 계산
private static final double EARTH_RADIUS_KM = 6371.0;

public double distanceTo(double targetLat, double targetLng) {
    double dLat = Math.toRadians(targetLat - this.latitude);
    double dLng = Math.toRadians(targetLng - this.longitude);
    double a = Math.sin(dLat / 2) * Math.sin(dLat / 2)
        + Math.cos(Math.toRadians(this.latitude))
        * Math.cos(Math.toRadians(targetLat))
        * Math.sin(dLng / 2) * Math.sin(dLng / 2);
    double c = 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a));
    return EARTH_RADIUS_KM * c * 1000; // meters
}

public boolean isWithinRange(double lat, double lng) {
    if (this.radiusMeters == null) return true;
    return distanceTo(lat, lng) <= this.radiusMeters;
}

Growth — 이 프로젝트에서 다음 프로젝트로

SISC에서 처음 접하고 좋다고 느낀 것들을, 다음 프로젝트에서 직접 설계하여 팀에 제안·적용했다.

SISC에서 배움

ErrorCode / ApiResponse 체계

SISC에서 팀의 공통 ErrorCode enum과 ApiResponse 구조를 처음 접했다. 일관된 에러 응답이 프론트엔드 파싱을 얼마나 단순하게 만드는지 체감.

다음 프로젝트에서

WeFin: BusinessException + ErrorCode + GlobalExceptionHandler 패턴을 직접 설계하고, 에러 처리 가이드를 노션에 문서화하여 팀 6인에게 공유·적용.
Flyway: 모듈별 접두사(C/U/A/P/R) 기반 ErrorCode 체계를 설계하여 에러 코드 충돌 없이 각 팀원이 독립적으로 추가할 수 있게 구성.

SISC에서 배움

패키지 구조와 DTO 계층 분리

Layered Architecture로 Controller → Service → Repository를 분리하는 구조를 처음 실습했다. 계층별 역할 분리의 중요성을 체감.

다음 프로젝트에서

WeFin: domain/web/global 3계층 분리를 제안. "Service가 web의 DTO를 import하면 의존 방향이 꼬인다"는 문제를 발견하고, Request/Response/Command/Info 4종 DTO 분리 규칙을 노션에 문서화하여 팀에 설득·적용.

SISC에서 배움

테스트 코드 작성

JUnit 5 + Mockito로 단위 테스트를 처음 작성했다. LocationTest에서 Haversine 계산 정확도를 검증하며, 테스트가 코드 신뢰도를 얼마나 높이는지 체감.

다음 프로젝트에서

WeFin: OrderServiceTest, PortfolioServiceTest 등 비즈니스 로직 테스트를 BDDMockito로 작성. 새 기능마다 테스트 코드를 함께 작성하는 습관.
Flyway: SafeCount 부분 실패 테스트, 스케줄러 예외 복원력 테스트 등 엣지 케이스까지 커버.

SISC에서 배움

협업 방식

User 모듈과 @ManyToOne FK로 직접 연결하여 개발했다. 통합 시 엔티티 변경이 상대 모듈에 영향을 주는 결합도 문제를 경험.

다음 프로젝트에서

WeFin: MarketPriceProvider, StockInfoProvider 인터페이스 계약 + ID 참조 방식으로 모듈 간 결합도를 낮춤. Mock으로 독립 개발 → 머지 후 통합하는 워크플로우 도입.
