Calculation

금액 · 수수료 계산

총액과 수수료를 산출. 수수료는 금융 관례에 따라 절사(DOWN).

totalAmount = price × quantity fee = totalAmount × 0.00015 (절사)

Pessimistic Lock

예수금 차감

findByIdForUpdate()로 비관적 락 획득 후 잔고 검증 → 차감. DB 락 + 엔티티 검증 이중 방어.

차감액 = totalAmount + fee

Persistence

Order 생성 · Trade 생성

Order(PENDING) 저장 → Trade.createBuyTrade()로 체결 기록 생성 (tax=0) → order.fill(quantity)로 FILLED 전이.

Pessimistic Lock

포트폴리오 갱신

addHolding() — 기존 보유 시 수량 추가 + 평균단가 재계산, 신규면 Portfolio 생성. totalBuyAmount로 소수점 오차 방지.

Event (AFTER_COMMIT)

체결 이벤트 발행

OrderMatchedEvent.ofBuy() 발행. 트랜잭션 커밋 후에만 리스너 실행 → WebSocket 알림, 채팅 등 후속 처리.

Validation

수량 검증 + 현재가 조회

수량 검증 후 MarketPriceProvider로 현재가 조회.

Pessimistic Lock

보유 종목 확인

getPortfolioForUpdate()로 비관적 락 획득 → 보유 수량 ≥ 주문 수량 검증. 읽기 시점부터 락을 걸어 stale read 방지.

Calculation

금액 · 수수료 · 세금 · 실현손익 계산

매도는 수수료 + 세금(매도만 과세). 실현손익은 음수(손실)도 정상.

fee = totalAmount × 0.00015 (절사) tax = totalAmount × 0.0018 (절사, 매도만) realizedProfit = (currentPrice - avgPrice) × qty

Persistence

Order 생성 · Trade 생성

Order 저장 → Trade.createSellTrade()로 체결 기록 생성 (fee, tax, realizedProfit 포함) → FILLED 전이.

Domain Update

포트폴리오 차감

deductQuantity() — 수량 차감. 전량 매도 시 포트폴리오 행 삭제(hard delete).

Pessimistic Lock

예수금 입금 + 실현손익 누적

비관적 락으로 계좌 조회 → 매도 대금 입금 → 실현손익 누적.

입금액 = totalAmount - fee - tax

Event (AFTER_COMMIT)

체결 이벤트 발행

OrderMatchedEvent.ofSell() 발행. tax, realizedProfit 포함. 수익 ±50만 이상 시 채팅 알림 트리거.

Key Decisions

01

Order/Trade 1:N 분리 설계

Problem

주문과 체결이 항상 1:1이 아니다. 지정가 주문은 여러 번에 걸쳐 부분 체결될 수 있는데, 주문 테이블에 체결 정보를 합치면 부분체결 이력을 담을 수 없다.

Decision

Order와 Trade를 별도 엔티티로 분리하고, JPA 연관관계 대신 ID 참조(Long 필드)로 연결. Trade에 virtualAccountId, stockId를 비정규화 저장하여 JOIN 없이 독립 조회 가능하게 했다.

Result

부분체결 시나리오 정상 대응. Trade 조회 시 Order JOIN 불필요. 동시성 제어 범위가 각 엔티티 독립적 — 락 범위 최소화.

// Trade.java — 정적 팩토리로 매수/매도 분리
public static Trade createBuyTrade(...)  { ... tax=0, profit=null }
public static Trade createSellTrade(...) { ... tax, profit 포함 }

관련 블로그

02

비관적 락(PESSIMISTIC_WRITE) + 이중 방어

Problem

동시에 2명이 같은 계좌의 예수금을 차감하면 레이스 컨디션 발생. 잔고 1,000만원에서 동시에 800만원씩 차감하면 둘 다 성공해버린다.

Decision

Repository에 @Lock(PESSIMISTIC_WRITE) + findByIdForUpdate() 메서드 분리. 엔티티의 deduct()에서 잔고 부족 검증 — DB 락 + 애플리케이션 검증 2중 보호. 트레이딩 시스템은 고경합 환경이므로 낙관적 락의 재시도 비용보다 비관적 락이 효율적.

Result

동시성 테스트 통과 (잔고 음수 0건). 락 순서 규칙(Account → Portfolio → Order)으로 데드락 방지.

@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM VirtualAccount a WHERE a.virtualAccountId = :id")
Optional<VirtualAccount> findByIdForUpdate(@Param("id") Long id);

// 엔티티 레벨 이중 방어
public void deduct(BigDecimal amount) {
    if (this.balance.compareTo(amount) < 0)
        throw new BusinessException(ORDER_INSUFFICIENT_BALANCE);
    this.balance = this.balance.subtract(amount);
}

관련 블로그

03

totalBuyAmount 별도 저장으로 소수점 오차 방지

Problem

추가 매수 시 평균단가를 재계산할 때, avgPrice만 저장하면 이전 반올림된 값으로 역산하게 되어 소수점 오차가 누적된다.

Decision

totalBuyAmount를 엔티티 필드로 별도 저장. 추가 매수 시 원본 금액을 직접 누적하고, avgPrice는 항상 원본 기반으로 재계산.

Result

10번 이상 추가 매수해도 소수점 오차 0원. 매도 시 avgPrice는 변경하지 않음 — 남은 주식의 매수 단가는 불변.

public void addQuantity(Integer quantity, BigDecimal price) {
    this.totalBuyAmount = this.totalBuyAmount
        .add(price.multiply(BigDecimal.valueOf(quantity)));
    this.quantity += quantity;
    this.avgPrice = this.totalBuyAmount
        .divide(BigDecimal.valueOf(this.quantity), 2, RoundingMode.HALF_UP);
}

관련 블로그

04

이벤트 기반 아키텍처 (AFTER_COMMIT)

Problem

체결 완료 후 WebSocket 알림, 채팅 메시지, 랭킹 갱신 등 후속 처리가 필요. OrderService에서 직접 호출하면 강한 결합 + 새 기능 추가마다 수정 필요.

Decision

@TransactionalEventListener(AFTER_COMMIT)으로 트랜잭션 커밋 후에만 이벤트 실행. OrderService는 OrderMatchedEvent만 발행하고, 리스너에서 후속 처리.

Result

OrderService의 의존성에서 WebSocket/채팅/랭킹 서비스 제거. 새 기능 추가 시 리스너만 추가하면 됨 (OCP). 체결 실패 시 알림 미발행 — 데이터 정합성 보장.

// OrderService — 이벤트 발행만
eventPublisher.publishEvent(OrderMatchedEvent.ofBuy(...));

// Listener — 커밋 후에만 실행
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handleOrderMatchedEvent(OrderMatchedEvent event) { ... }

관련 블로그

05

엔티티 비즈니스 메서드 캡슐화 (@Setter 제거)

Problem

@Setter로 필드를 직접 변경하면 잔고 검증 같은 비즈니스 규칙이 무시될 수 있다. 또한 JPA 연관관계(@ManyToOne)를 사용하면 비관적 락 시 부모까지 로딩되어 락 범위가 확장된다.

Decision

@Setter 제거, deduct(), deposit(), addQuantity() 등 비즈니스 메서드로만 상태 변경. Trade는 updatable=false로 불변 설계.

Result

비즈니스 규칙이 엔티티에 캡슐화 — 서비스 레이어에서 규칙 누락 불가. 각 엔티티가 독립적으로 비관적 락 획득 가능.

Troubleshooting

DDL-엔티티 PK 필드명 불일치로 CI 전체 실패

증상: CI에서 전체 테스트 실패. DDL의 컬럼명과 JPA 엔티티 필드명이 매칭되지 않아 테이블 생성 자체가 실패.

원인: DDL에서 PK를 order_id로 정의했는데, 엔티티에서는 id로 선언. Spring naming strategy가 orderId → order_id로 변환하는데, 필드명이 id면 id로 매핑되어 불일치.

해결: 엔티티 PK 필드를 id → orderId로 변경. 이후 "DDL 우선 확인" 규칙을 팀 컨벤션에 추가.

교훈: DDL이 진실의 원천(source of truth). 팀 프로젝트에서 DDL은 여러 팀원이 공유하는 계약이므로, 엔티티 변경이 DDL 변경보다 안전하다.

매도 시 비관적 락 누락으로 stale avgPrice 읽기

증상: 매도 실현손익 계산에서 간헐적으로 잘못된 평균단가로 손익이 계산됨. 동시에 추가 매수와 매도가 발생하면 stale read.

원인: 매도 흐름에서 일반 조회 후 avgPrice를 읽고, 이후 deductQuantity 호출. 조회~변경 사이에 다른 트랜잭션이 addQuantity를 실행하면 이미 읽은 값은 반영되지 않음.

해결: 매도 시 포트폴리오 조회를 findByVirtualAccountIdAndStockIdForUpdate()로 변경. 조회 시점부터 변경 완료까지 전체 구간에 락 유지.

교훈: 금액 관련 로직에서는 읽기부터 락을 걸어야 한다. 읽은 값이 다른 연산의 입력값이 되는 경우, 조회 시점에 이미 정합성이 깨질 수 있다.

트랜잭션 커밋 전 이벤트 발행으로 정합성 깨짐

증상: 체결 과정에서 예외가 발생해 롤백됐는데, WebSocket으로 "체결 완료" 알림이 이미 전송됨.

원인: 초기 @EventListener는 이벤트 발행 시점(트랜잭션 내부)에 즉시 실행. 이벤트 핸들러 실행 후 트랜잭션이 롤백될 수 있음.

해결: @EventListener → @TransactionalEventListener(AFTER_COMMIT) 변경. 트랜잭션이 정상 커밋된 후에만 핸들러 실행.

교훈: 이벤트 기반 아키텍처에서 "언제 발행하느냐"가 "무엇을 발행하느냐"만큼 중요하다. 금융 시스템에서는 데이터가 확정된 후에만 외부에 알려야 한다.

Collaboration — 시세/종목 담당 팀원과의 협업

문제

서로의 모듈이 동시에 개발 중 — 어떻게 독립적으로 진행할 것인가?

나(주문/체결)는 현재가와 종목 정보가 필요하고, 팀원 B(시세/종목)는 체결 이벤트가 필요했다. 하지만 서로의 코드가 아직 없는 상태에서 개발을 시작해야 했다.

Interface Contract

인터페이스 계약 먼저 정의

MarketPriceProvider, StockInfoProvider, ExchangeRateProvider 3개 인터페이스를 domain/trading/common/에 정의. "무엇을 줄 것인가"만 합의하고, "어떻게 구현할 것인가"는 각자 결정. 

// A가 정의, B가 구현
public interface MarketPriceProvider {
    BigDecimal getCurrentPrice(String stockCode);
    String getCurrency(String stockCode);
}

public interface StockInfoProvider {
    Stock getStock(Long stockId);
    String getStockName(String stockCode);
}

Independent Dev

Mock으로 각자 독립 개발

A는 테스트에서 MarketPriceProvider를 Mock으로 주입하여 B의 구현 없이 OrderService를 개발·테스트. B는 한투 Open API 연동에 집중. 

// OrderServiceTest — B 없이 단위 테스트 가능
given(marketPriceProvider.getCurrentPrice("005930"))
    .willReturn(new BigDecimal("70000"));
given(stockInfoProvider.getStock(1L))
    .willReturn(mockStock);

Integration

B의 구현 머지 후 통합

B의 MarketService implements MarketPriceProvider, StockService implements StockInfoProvider 머지 후, OrderService의 파라미터를 stockCode → stockId로 변경. 통합 테스트로 연동 검증.

Shared Rules

공유 자원 충돌 방지 규칙

ErrorCode는 각자 prefix 범위에서만 추가 (ORDER_*는 A, MARKET_*는 B). 같은 파일 수정 시 먼저 merge한 사람 기준, 나중 사람이 rebase. 비관적 락 순서(Account → Portfolio → Order) 규칙을 B의 지정가 체결 엔진에도 적용.

Team Impact

  • API 응답 규격 통일 — ApiResponse<T> 구조를 제안하여 팀 전체(6인) 적용. 프론트엔드 파싱 로직 단일화. 블로그
  • ErrorCode 체계 설계 — 도메인별 prefix (ORDER_*, MARKET_*)로 에러 코드 구분. 숫자 대신 의미 있는 이름 사용.
  • 에러 처리 가이드 작성 — BusinessException + ErrorCode + GlobalExceptionHandler 패턴으로 팀 전체 에러 처리 통일. Service에서 throw만 하면 자동 응답되는 구조를 설계하고 노션에 문서화. 노션 문서
  • DTO 계층 분리 가이드 작성 — 패키지를 domain/web/global로 나누자고 제안하는 과정에서 "Service가 web의 DTO를 import하면 의존 방향이 꼬인다"는 문제를 발견. 직접 학습 후 Request/Response/Command/Info 4종의 분리 규칙을 노션에 문서화하여 팀을 설득·적용. 노션 문서
  • A-B 인터페이스 계약 — MarketPriceProvider, StockInfoProvider 인터페이스 정의. 상대 모듈 미완성 시에도 독립 개발 가능.
  • 코딩 컨벤션 문서화 — 패키지 규칙, 네이밍 규칙, DB 컨벤션, 락 순서 등을 문서(trading-convention.md)로 정리·관리.
